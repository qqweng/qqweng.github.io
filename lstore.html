L-Store Implementation

Lineage-based Data Store (L-Store) is a relational database that combines the real-time processing of transactional and analytical workloads in a lineage-based storage architecture. This allows for contention-free and lazy staging of columnar data from a write-optimized form into a read-optimized form that supports querying and retaining both current and historic data. Milestone 1

The 3 key stages for Milestone 1 are:

  1. Data model
  2. Bufferpool Management
  3. Query Interface

Implementation

The Database class is the general interface that handles: starting and shutting down of the database instance.


The Query class provides standard SQL operations: insert, select, update, delete, and sum. This class completes the query interface between the user and the table.


The Table class is the core of our database. We are able to view and manage records in the table. We used dictionary to hold the location of each record and it's corresponding unique identifier (RID). We have incorporated the buffer management for this milestone in the table class's page directory.


The Record class holds all of a records information, which is stored in the Table class. Most of the data model is completed in this class.


The Page class handles low-level physical storage. This class is currently used by the Record class to store and retrieve pages, split into base pages and tail pages.

Milestone 2

The 3 key stages for Milestone 2 are:

  1. Durability & Bufferpool Extension
  2. Data Reorganization
  3. Indexing

Implementation

The Database class has a new additional change of providing a fixed constant number of pages the bufferpool will have.


The Bufferpool class will handle the replacement policy of the pages that exceed the pages the bufferpool is set to have. The pinning and storing of old pages to disk is handled here using the Pickle library. This gives our database durability & the bufferpool extension.


The Table class handles the data reorganization by contention-free merging the tail records and updating the base record with the latest information by using a deque so that we can store the queue if table closes in the middle of merging


The Index class allows the table to process queries faster by indexing all the columns. We used a B-Tree to hold all the indexes.

Milestone 3

The 2 key stages for Milestone 3 are:

  1. Transaction Semantics
  2. Concurrency Control

Implementation

The Lock Manager class contains the locks for all base records. This ensures that multiple transactions don't write to the same record. A record has one X-lock for writing and multiple S-locks for reading.


The Transaction class handles the multithreading concurrency control of the multiple transaction workers. Only when all the transaction workers finish does the table commit and save the changes. Otherwise, the changes are aborted and fails to pass.


The Transaction Worder class does all the work as a single thread that works with all the other existng transaction workers. All the transactions that this worker has to do is also unpacked in this class.
